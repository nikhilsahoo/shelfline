# MVP 2 Design

## Goal

MVP 2 turns epub-tui from a functional catalog-first downloader into a polished terminal reading and library application. The main outcome is a better everyday loop: browse catalogs, download books, find books locally, read EPUBs comfortably, and resume where the user left off.

The milestone should keep the strong MVP 1 foundation: Python, Textual/Rich, OPDS 1.x, one active download, local SQLite metadata, JSON user configuration, and Windows/Linux support.

## Product Theme

MVP 2 theme: polished terminal reading and library experience.

The app should feel like a real book app that happens to live in the terminal, not a sequence of plain text debug screens. Visual polish matters, but it should support repeated use: fast scanning, predictable navigation, clear state, and readable long-form text.

## Scope

Included in MVP 2:

- A cohesive Textual app shell with consistent navigation, content, detail, and status regions.
- A better visual language for catalog, library, entry, download, and reader screens.
- Scrollable EPUB reader screen for downloaded EPUB files.
- Chapter/section navigation for EPUBs.
- Reading progress persistence for EPUBs.
- Simple bookmarks for EPUBs.
- Library search and filtering by title, author, catalog, format, and read/progress state.
- Better library rows with metadata, progress, and read-state signals.
- Keyring-backed Basic Auth password storage where available, with an explicit fallback path.
- Updated documentation and tests for the new user flows.

Explicitly out of scope for MVP 2:

- OPDS 2.x JSON feeds.
- PDF, DjVu, CBR, CBZ, or other in-terminal rendering.
- Multiple simultaneous downloads or a download queue.
- OAuth, browser login, cookies, bearer tokens, or custom authentication flows.
- Full annotations, highlights, cloud sync, multi-device sync, and collections/tags.
- Full-text search inside book contents.

## User Experience

MVP 2 keeps keyboard-first operation while making every screen more readable and self-explanatory.

The app shell should provide:

- Header: app title, current area, and compact active context.
- Navigation/sidebar: Catalogs, Library, Reader, and Settings/Auth where relevant.
- Main content area: list, feed, reader, or form.
- Detail panel: selected item metadata, cover fallback, acquisition choices, progress, or help text.
- Footer/status bar: key hints, busy/error/success state, and network/download progress.

The layout should adapt to narrow terminals. On small widths, sidebar/detail regions may collapse into stacked sections or simplified status lines. Text must not overlap or rely on fixed wide terminals.

## Visual Design

The UI should use Textual CSS and Rich styling instead of plain stacked strings.

Required polish:

- Clear selected and focused states.
- Dense but legible list rows.
- Consistent screen titles and section labels.
- Distinct styles for folders, books, unsupported formats, errors, warnings, success, and progress.
- Empty states that explain what action is available next.
- Loading states that remain visible for outgoing calls.
- A consistent key-hint footer across screens.
- A restrained default theme that works on dark terminals.

The design should avoid decorative noise. It should feel like a quiet reading and library tool: structured, calm, and practical.

## Navigation Model

Global navigation:

- `c`: Catalogs
- `l`: Library
- `R`: resume the most recently opened reader item when available
- `r`: refresh where the current screen supports refresh
- `s`: Search/filter where the current screen supports it.
- `b`: Back
- `q`: Quit or close current modal/screen

List navigation:

- `j` / `down`: next item
- `k` / `up`: previous item
- `enter`: open selected item
- `/`: focus filter/search input where available

Reader navigation:

- `j` / `down`: scroll down
- `k` / `up`: scroll up
- `space`: page down
- `shift+space` or `pageup`: page up
- `n`: next chapter/section
- `p`: previous chapter/section
- `m`: add/remove bookmark at current reader position
- `g`: show section/bookmark navigator

Exact bindings may be adjusted during implementation if Textual conflicts occur, but the final app must document and display the active bindings.

## Reader

The MVP 2 reader upgrades the MVP 1 static preview into an interactive EPUB reading screen.

Reader behavior:

- Open a downloaded EPUB from the library.
- Render extracted section text in a scrollable view.
- Show current title, current section/chapter, and progress.
- Navigate between sections.
- Persist reading progress automatically.
- Resume from the last saved position when reopening the book.
- Add and remove simple bookmarks.
- Show a lightweight bookmark/section navigator.
- Fail gracefully for malformed EPUBs and unsupported formats.

Progress model:

- Store book identity using the local file path plus stable metadata where available.
- Track current section index and an approximate intra-section scroll offset or paragraph index.
- Store last-opened timestamp.
- Keep read/unread state compatible with MVP 1.

Bookmarks:

- Store section index, optional intra-section offset, label/title, and created timestamp.
- Bookmarks are local-only in MVP 2.
- Bookmark labels can default to section title and a short text snippet.

## Library

The library becomes the main local management surface.

Library requirements:

- Search/filter by title and author.
- Filter by catalog, format, read/unread, in-progress, and completed.
- Show progress/read-state in each row.
- Open EPUBs into the reader.
- Show unsupported-format message for non-EPUB files.
- Refresh from repository.
- Keep existing mark read/unread and delete behavior.

Suggested row content:

- Title
- Author(s)
- Format
- Source catalog
- Progress or read state
- Local file existence state

The search/filter UI should be fast and local. It should query existing SQLite metadata and not scan full book contents.

## Catalog Experience

Catalog browsing remains OPDS 1.x and should benefit from the new app shell.

Catalog requirements:

- Keep folder/book labels and breadcrumbs from MVP 1.
- Preserve Basic Auth support.
- Preserve single-download behavior.
- Render selected entry details in a detail panel rather than only inline text.
- Keep visible busy/progress indicators for fetch, navigation, and download operations.
- Keep recoverable errors visible with back/retry paths.

Catalog search is not required unless a catalog exposes OPDS search links and implementation remains small. If catalog search expands the scope, defer it.

## Credentials

MVP 2 should stop treating password-in-JSON as the preferred path.

Credential requirements:

- Store catalog metadata in JSON as before: name, URL, and non-secret settings.
- Store Basic Auth passwords in an OS keyring where available.
- Keep Basic Auth usernames in JSON so the file remains understandable and editable; keep passwords in keyring where available.
- Provide a migration path for existing JSON `auth.password` values.
- Preserve a clear fallback for environments without keyring support.
- Never display passwords in status, logs, errors, exported metadata, or cached feeds.

Fallback options:

- Recommended: keep JSON password support as an explicit fallback with a warning in docs.
- Alternative: prompt every session when keyring is unavailable and password is missing.

The implementation plan should evaluate the current `keyring` Python package and its Windows/Linux behavior before choosing the final fallback mechanics.

## Data Model

SQLite remains the app-managed state store.

MVP 2 adds:

- `reading_progress`: local file path or book id, section index, intra-section position, updated timestamp.
- `bookmarks`: local file path or book id, section index, intra-section position, label, created timestamp.
- Search/filter helper indexes for title, author, catalog, format, and read/progress state if current query performance is not adequate.
- Credential reference fields only if keyring integration needs stable service/account IDs.

Config JSON remains human-editable and should stay small. It should not become a general database.

## Architecture

MVP 2 should keep the layered shape:

- `tui`: app shell, styled widgets, screens, key bindings, layout, status handling.
- `reader`: EPUB extraction and reader document model.
- `library`: book metadata, reading progress, bookmarks, search/filter queries.
- `config`: JSON configuration and credential metadata.
- `credentials`: keyring/fallback abstraction for Basic Auth secrets.
- `catalog`, `downloads`, and `services`: preserve MVP 1 behavior with small adaptations for the shell.

The UI refresh should avoid turning `screens.py` into a large all-purpose file. If the implementation grows, split reusable widgets and screen classes into smaller modules under `src/epub_tui/tui/`.

## Error Handling

MVP 2 should keep errors visible and recoverable.

Required cases:

- Reader cannot parse EPUB.
- Reading progress/bookmark write fails.
- Library search query fails.
- Local file is missing.
- Keyring unavailable or locked.
- Credential migration fails.
- Catalog fetch/download errors from MVP 1.

Errors should include the action that failed and the next available action, such as back, retry, reconfigure credentials, or return to library.

## Testing

Core tests:

- Reading progress save/load/update.
- Bookmark add/remove/list.
- Library search/filter combinations.
- Credential store abstraction with fake keyring backend.
- JSON credential migration and fallback behavior.
- Reader resume behavior for EPUBs.

TUI tests:

- App shell renders expected regions.
- Catalog, library, entry, download, and reader screens show key hints.
- Library search filters visible rows.
- Reader scroll and section navigation update visible state.
- Reader resume opens at saved progress.
- Bookmark action creates/removes a bookmark.
- Keyring unavailable path shows recoverable status.

Verification:

- Full test suite on Windows and Linux.
- `python -m epub_tui --help`.
- Manual smoke: catalog browse, download, library search, open EPUB, scroll, save progress, reopen, bookmark, delete book.

## Implementation Strategy

Recommended order:

1. App shell and TUI styling foundation.
2. Library search/filter and improved library rows.
3. Reading progress data model.
4. Scrollable EPUB reader and section navigation.
5. Bookmarks.
6. Credential storage abstraction and keyring integration.
7. Documentation, final manual smoke checklist, and release verification.

The UI shell should come first so later reader and library features land into the intended structure instead of being bolted onto the MVP 1 screens.

## Success Criteria

MVP 2 is complete when:

- The app has a cohesive, styled Textual interface rather than plain stacked text screens.
- A user can search their local library and open an EPUB into a scrollable reader.
- Reading progress is saved and restored.
- A user can add/remove bookmarks.
- Basic Auth passwords can be stored outside JSON when keyring support is available.
- MVP 1 catalog/download/library flows still work.
- Full automated tests pass.
- Windows and Linux manual smoke checks pass.

## Deferred After MVP 2

- OPDS 2.x JSON feeds.
- PDF/DjVu/CBR/CBZ in-terminal rendering.
- Download queue.
- Full-text search inside books.
- Annotations and highlights.
- Sync and multi-device state.
- Collections, tags, and bulk library edits.
